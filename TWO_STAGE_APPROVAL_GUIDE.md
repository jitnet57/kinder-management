# AKMS 2단계 승인 프로세스 가이드

## 📋 개요

AKMS는 강화된 보안을 위해 **2단계 승인 프로세스**를 적용합니다:

1. **1단계: 관리자 승인** (Admin Approval)
   - 시스템 관리자가 사용자/디바이스 정보를 검증
   - 기본적인 보안 확인

2. **2단계: 개발자 최종 승인** (Developer Approval)
   - 개발자(시스템 소유자)가 최종 승인
   - 최종 접근 권한 부여

---

## 🔄 승인 프로세스 흐름도

```
신규 사용자 로그인
    ↓
디바이스 자동 등록
    ↓
대기 상태 (pending)
    ↓
┌─────────────────────────────┐
│  1단계: 관리자 승인           │
│  Admin Approval             │
│ ✅ 승인 or ❌ 거부           │
└─────────────────────────────┘
    ↓ (승인)
┌─────────────────────────────┐
│  상태: admin_approved       │
│ (관리자 승인됨)              │
│ (개발자 승인 대기)           │
└─────────────────────────────┘
    ↓
┌─────────────────────────────┐
│  2단계: 개발자 승인           │
│  Developer Approval         │
│ ✅ 승인 or ❌ 거부           │
└─────────────────────────────┘
    ↓ (승인)
┌─────────────────────────────┐
│  상태: developer_approved   │
│ (최종 승인됨)               │
│ 시스템 접근 가능 ✅          │
└─────────────────────────────┘
```

---

## 👥 역할 정의

### 1. 관리자 (Admin)
```
역할:   시스템 관리자
권한:   1단계 승인 권한
URL:    /admin/approvals
확인:   사용자 정보, IP 주소, 디바이스 정보
```

**책임:**
- 신규 사용자/디바이스 등록 정보 검증
- 부적절한 접근 시도 거부
- 보안 기본 사항 확인

### 2. 개발자 (Developer)
```
역할:   시스템 소유자 / 개발자
권한:   2단계 최종 승인 권한
URL:    /admin/approvals (역할 자동 감지)
확인:   최종 검증 및 승인
```

**책임:**
- 관리자가 승인한 사용자/디바이스 최종 검증
- 비즈니스 규칙 확인
- 최종 접근 권한 부여

## 📱 사용자 상태 변화

### 사용자 상태

```
pending
    ↓ (관리자 승인)
admin_approved
    ↓ (개발자 승인)
developer_approved  ← 접근 가능 ✅
    ↓ (거부)
suspended
```

### 상태별 설명

| 상태 | 관리자 | 개발자 | 접근 | 설명 |
|------|--------|--------|------|------|
| pending | ❌ | ❌ | ❌ | 대기 중 |
| admin_approved | ✅ | ❌ | ❌ | 관리자 승인됨, 개발자 대기 |
| developer_approved | ✅ | ✅ | ✅ | 최종 승인됨 |
| suspended | ❌/✅ | ❌/✅ | ❌ | 거부됨 (접근 불가) |

## 🖥️ 관리자 승인 인터페이스 사용법

### 페이지 접속

```
1. 로그인 후 대시보드
2. 왼쪽 사이드바: "👨‍💼 승인관리" 클릭
3. 또는 URL: /admin/approvals
```

### 필터링

```
⏳ 대기중          - pending 상태 (아직 승인 전)
📋 관리자 승인됨   - admin_approved 상태 (개발자 승인 대기)
✅ 최종 승인됨     - developer_approved 상태 (완료)
📋 전체            - 모든 사용자/디바이스
```

### 사용자 승인 프로세스

#### Step 1: 대기중 사용자 확인
```
필터: ⏳ 대기중 선택
→ 승인 대기 중인 사용자 목록 표시

각 사용자 정보:
├─ 이메일: user@example.com
├─ 이름: 김철수
├─ 역할: therapist
├─ 등록일: 2024년 4월 26일
└─ 상태: 
   ├─ 관리자 ⏳ (미승인)
   └─ 개발자 ⏳ (대기)
```

#### Step 2: 관리자로서 정보 검증
```
확인 사항:
□ 이메일이 올바른가?
□ 사용자를 알고 있는가?
□ IP 주소가 예상되는 범위인가?
□ 등록 시간이 합리적인가?
□ 추가 보안 확인 필요한가?
```

#### Step 3: 관리자 승인 버튼 클릭
```
1단계: 관리자 승인 섹션
├─ [✅ 관리자 승인] 버튼 클릭
└─ [❌ 거부] 버튼

결과: 상태 → admin_approved
      (개발자 최종 승인 섹션 표시)
```

#### Step 4: 개발자로서 최종 승인
```
2단계: 개발자 최종 승인 섹션
(관리자 승인 후 표시됨)

├─ [✅ 최종 승인] 버튼 클릭
└─ [❌ 거부] 버튼

결과: 상태 → developer_approved
      (접근 가능) ✅
```

## 📋 디바이스 승인 프로세스

디바이스 승인은 사용자 승인과 동일한 2단계 프로세스:

```
Device Status Flow:

  ↓ (개발자 승인)
developer_approved ← 접근 가능 ✅
  ↓ (거부)

```

```
등록일:      2024년 4월 26일 14:30
```

## 🔐 보안 체크리스트

### 관리자 승인 시 확인

**사용자 검증**
- [ ] 이메일 주소가 유효한가?
- [ ] 이메일 도메인이 신뢰할 수 있는가?
- [ ] 사용자를 개인적으로 알고 있는가?
- [ ] 최근 가입한 직원이나 인수한 파트너인가?

**디바이스 검증**
- [ ] IP 주소가 회사 범위 또는 알려진 위치인가?
- [ ] 브라우저 정보가 정상인가?
- [ ] 운영체제가 예상되는 것인가?
- [ ] 등록 시간이 업무 시간인가?

### 개발자 최종 승인 시 확인

**비즈니스 규칙 확인**
- [ ] 사용자 역할이 적절한가?
- [ ] 부서/팀이 일치하는가?
- [ ] 디바이스 수가 합리적인가?
- [ ] 추가 보안 교육이 필요한가?

**최종 결정**
- [ ] 관리자 승인이 완료되었는가?
- [ ] 모든 정보가 정확한가?
- [ ] 승인해도 되는 사용자/디바이스인가?

## 📊 승인 통계 및 현황

### 대시보드 (향후 추가)

```
승인 현황:
├─ 대기중: 5명
├─ 관리자 승인: 2명 (개발자 대기)
├─ 최종 승인: 23명
└─ 거부됨: 1명
```

### 필터별 조회

```
필터: ⏳ 대기중
→ 관리자 승인이 필요한 목록

필터: 📋 관리자 승인됨
→ 개발자 최종 승인이 필요한 목록

필터: ✅ 최종 승인됨
→ 접근 가능한 사용자/디바이스 목록
```

## ⏱️ 승인 처리 시간 가이드

| 단계 | 담당자 | 예상 시간 | 설명 |
|------|--------|---------|------|
| 1단계 | 관리자 | 즉시~5분 | 기본 정보 검증 |
| 2단계 | 개발자 | 즉시~15분 | 최종 검증 및 승인 |
| **전체** | 관리자+개발자 | **즉시~20분** | 신규 사용자 등록 완료 |

## 🚨 거부 시나리오

### 관리자가 거부하는 경우

```
거부 사유 예시:
- 알 수 없는 IP 주소
- 의심스러운 이메일 도메인
- 비정상적인 접근 시간
- 사용자를 모르는 경우
- 보안 정책 위반
```

**결과:**
- 사용자 상태 → suspended
- 해당 계정 접근 불가능
- 다시 신청 필요

### 개발자가 거부하는 경우

```
거부 사유 예시:
- 부서/팀이 맞지 않음
- 사용자 역할이 부적절함
- 비즈니스 규칙 위반
- 추가 보안 확인 필요
- 계약 상태 미확인
```

**결과:**
- 사용자 상태 → suspended
- 해당 계정 접근 불가능
- 사용자가 관리자에게 문의 필요

## ❓ FAQ

### Q1: 거부된 사용자는 다시 신청할 수 있나요?
**A:** 네, 거부된 사용자는:
1. 새로운 이메일로 다시 가입
2. 또는 관리자/개발자에게 문의하여 계정 재활성화

### Q2: 한 사람이 여러 역할을 할 수 있나요?
**A:** 현재는:
- 각 계정이 하나의 역할 (admin/therapist/parent)
- 향후 다중 역할 지원 가능

### Q3: 승인 거부 후 몇 일 뒤에 다시 신청할 수 있나요?
**A:** 제한 없음. 즉시 다시 신청 가능.

### Q4: 개발자 승인 없이 관리자 승인만으로 접근 가능한가요?
**A:** 아니요. 반드시:
- 관리자 승인 ✅
- AND 개발자 승인 ✅
- 둘 다 필요함

### Q5: 관리자와 개발자가 같은 사람이면 어떻게 되나요?
**A:** 같은 사람이면:
- 1단계에서 자신이 승인 (관리자로)
- 2단계에서 자신이 승인 (개발자로)
- 두 번 클릭하면 됨

## 📞 문제 해결

### 승인 페이지가 보이지 않음

```
확인사항:
1. 로그인 상태 확인
2. 사용자 역할이 admin인지 확인
3. 브라우저 캐시 삭제 (Ctrl+Shift+Delete)
4. /admin/approvals URL 직접 접속
```

### 개발자 승인 섹션이 보이지 않음

```
원인: 관리자 승인이 완료되지 않음

해결:
1. 필터를 "📋 관리자 승인됨"으로 변경
2. 해당 사용자/디바이스 확인
3. 관리자로서 1단계 승인 먼저 완료
```

### 승인 후에도 접근 불가

```
확인사항:
1. 2단계 모두 완료되었는지 확인
   - 관리자 ✅
   - 개발자 ✅
2. 브라우저 캐시 삭제
3. 다시 로그인 시도
4. localStorage에 데이터 저장되었는지 확인
```

## 🎓 운영 팁

### Best Practice

```
1. 정기적으로 승인 현황 검토
   └─ 주 1회 /admin/approvals 방문

2. 거부된 사용자 추적
   └─ 거부 사유 메모 작성

3. 승인 시간 모니터링
   └─ 병목 지점 확인

4. 보안 교육 후 승인
   └─ 사용자 교육 → 관리자 승인 → 개발자 승인
```

### 운영 일정

```
매일:
├─ 아침: 대기중 신규 사용자 확인
└─ 저녁: 승인 결과 확인

주 1회 (금요일):
└─ 관리자/개발자 승인 현황 검토

월 1회:
└─ 승인 정책 및 프로세스 검토
```
